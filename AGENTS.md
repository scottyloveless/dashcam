# AGENTS.md - Guidelines for AI Agents

This document provides instructions and conventions for AI coding agents working in this repository.

## Project Overview

**dashcam** is an IT infrastructure monitoring tool written in Go that collects metrics from network devices and stores them in TimeScaleDB (PostgreSQL).

- **Go Version**: 1.25.0
- **Database**: PostgreSQL 17 with TimeScaleDB
- **Key Libraries**: pgx/v5, sqlc, goose, prometheus-community/pro-bing, httprouter

## Build/Lint/Test Commands

### Running the Application

```bash
# Run the collector service
go run ./cmd/collector/

# Run the web server
go run ./cmd/webserver/

# Build both binaries
go build -o bin/collector ./cmd/collector/
go build -o bin/webserver ./cmd/webserver/
```

### Code Generation

Database queries are generated by **sqlc**. After modifying SQL files, regenerate:

```bash
sqlc generate
```

Database migrations use **goose**:

```bash
# Run migrations
goose -dir=sql/schema postgres "DATABASE_URL" up

# Rollback last migration
goose -dir=sql/schema postgres "DATABASE_URL" down

# Check migration status
goose -dir=sql/schema postgres "DATABASE_URL" status
```

### Running Tests

**Note**: Currently, there are no test files in this project. When adding tests:

```bash
# Run all tests
go test ./...

# Run tests with verbose output
go test -v ./...

# Run tests for a specific package
go test -v ./cmd/collector/
go test -v ./cmd/webserver/

# Run a single test
go test -v -run TestFunctionName ./...

# Run tests with coverage
go test -cover ./...
```

### Linting

Go formatting (automatic with `gofmt`):

```bash
# Format all Go files
gofmt -w .

# Check formatting without modifying
gofmt -d .
```

SQL linting with SQLFluff:

```bash
# Lint all SQL files
sqlfluff lint sql/

# Fix linting issues automatically
sqlfluff fix sql/
```

### Database Connection

Set `DATABASE_URL` in `.env` file:

```
DATABASE_URL="postgres://user:password@localhost:5432/dbname?sslmode=disable"
```

## Code Style Guidelines

### Go Conventions

1. **Imports**: Standard library first, then third-party, then internal packages.
   Blank line between groups.

```go
import (
    "context"
    "flag"
    "fmt"
    "log/slog"
    "time"

    "github.com/jackc/pgx/v5/pgxpool"
    "github.com/joho/godotenv"
    "github.com/scottyloveless/dashcam/internal/database"
)
```

1. **Naming Conventions**:
   - Types/Structs: `PascalCase` (e.g., `type application`, `type Trigger`)
   - Variables/Functions: `camelCase` (e.g., `pool`, `openDB`, `collectPing`)
   - Constants: `PascalCase` for public, `camelCase` for private (e.g., `SeverityEnumWarning`)
   - Database enums: `TypeNameEnum` prefix (e.g., `SeverityEnum`, `StateEnum`)

2. **Error Handling Pattern**:
   - Return errors from functions when operation fails
   - Log errors at the call site before taking action
   - Use `context.WithTimeout` for database operations
   - Use `defer cancel()` after `context.WithTimeout`

```go
// Good pattern
func openDB(cfg config) (*pgxpool.Pool, error) {
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    pool, err := pgxpool.New(ctx, cfg.db.dsn)
    if err != nil {
        return nil, err
    }
    defer pool.Close()

    err = pool.Ping(ctx)
    if err != nil {
        defer pool.Close()
        return nil, err
    }
    return pool, nil
}
```

1. **HTTP Handlers**: Attach to application struct as methods

```go
func (app *application) alertsPartialHandler(w http.ResponseWriter, r *http.Request) {
    alerts, err := app.queries.GetAlerts(r.Context())
    if err != nil {
        app.logger.Error(err.Error())
        return
    }
    // ...
}
```

1. **Logging**: Use `log/slog` for structured logging

```go
logger := slog.New(slog.NewTextHandler(os.Stdout, nil))
logger.Info("starting server", "addr", srv.Addr, "env", cfg.env)
logger.Error(err.Error())
```

1. **Context Propagation**: Always pass context to database operations

```go
func (app *application) collectPing(trigger Trigger) error {
    ctx := context.Background()
    // Use ctx for database calls
}
```

1. **Generated Code**: Do not edit files in `internal/database/` directly. They
   are generated by sqlc from SQL files in `sql/queries/`.

### SQL Conventions

1. **File Organization**:
   - Schema migrations: `sql/schema/###_name.sql` (numbered sequentially)
   - Query definitions: `sql/queries/*.sql`

2. **Naming**: `snake_case` for tables, columns, and query names

3. **Query Comments**: sqlc queries use `-- name: QueryName :type` format

```sql
-- name: GetDevices :many
SELECT
    id,
    nickname,
    ip_address
FROM devices
WHERE enabled = true
```

1. **Keywords**: UPPERCASE for SQL keywords (SELECT, FROM, WHERE, etc.)

2. **Migration Format**: Use goose migration syntax with `-- +goose Up` and `-- +goose Down`

```sql
-- +goose Up
CREATE TABLE devices (
    id UUID PRIMARY KEY,
    nickname TEXT NOT NULL
);

-- +goose Down
DROP TABLE devices;
```

### Additional Guidelines

1. **Configuration**: Use `flag` package for CLI flags. Load environment from `.env` using `godotenv`.

2. **Concurrency**: Use goroutines with context-aware operations. Use `defer ticker.Stop()` for cleanup.

3. **Database Connections**: Use connection pooling with pgxpool. Configure max connections via flags.

4. **No Comments**: Avoid adding unnecessary comments. Code should be self-documenting.

5. **Dependencies**: After adding dependencies, run `go mod tidy`.

6. **Gitignore**: The `.env` file is gitignored. Never commit secrets.
